Q: "How do you handle mid-sprint requirement changes?"
  • Acknowledge immediately — do not ignore or flat-refuse
  • Assess impact — does this affect stories already in development this sprint?
  • If yes — freeze sprint scope. Log as new story for next sprint. Explain trade-off: "Adding this means delaying Story X. Confirm?"
  • If small clarification — update AC in JIRA immediately and notify dev
  • Document every decision with date, stakeholder name, and reason
Q: "Developers keep asking you questions — doesn't that mean requirements were unclear?"
  • Honest answer: "Yes, sometimes it means the AC needed more depth. I treat every repeated question as a signal to improve the story template."
  • Update the AC in JIRA with the answer — not just in chat — so QA and future devs have the same source of truth
  • Pattern recognition: if the same question type keeps coming, add an explicit section to the story template going forward
  • Mohan's example: On PHNB, API error scenarios kept coming up. I added a standard integration error handling section to all API stories — timeout, null response, malformed JSON — each with explicit expected behaviour. Dev questions dropped significantly in the next sprint.
Q: "How do you manage refinement when stakeholders are too busy to attend?"
  • Pre-work: send story drafts 24 hours before — stakeholders review async, come prepared
  • Time-box: 60 minutes maximum — park stories not ready after 15 min of discussion
  • Agree upfront who can make requirement decisions in the room — avoid "I need to check with my manager" blocking the sprint
  • Escalate chronic unavailability to PM — this is a project risk, not a BA problem to absorb silently

3DS 2.0 — 60-Second Explainer for Any Interviewer
"3DS stands for 3-Domain Secure. The three domains are the Issuer bank, the Acquirer bank, and the Interoperability domain — Visa or Mastercard acting as the routing layer. When you shop online and get an OTP page — that's 3DS. In version 1.0, every transaction redirected through a browser page. In 2.0, the bank's Access Control Server receives over 100 data points — device fingerprint, transaction history, IP, merchant category — and decides in milliseconds: silently authenticate this (frictionless), or ask for OTP. For PNB, the platform delivering this was Wibmo's ACCOSA® IVS, paired with the TRIDENT Risk Engine."

My Signature Contribution — The Cross-System Counter
This is the most important thing I did on PHNB. Talk about this in every interview. It shows discovery depth, fraud domain knowledge, and the ability to make business risk tangible enough to drive engineering decisions.
THE GAP I FOUND
ACCOSA® IVS tracks wrong OTP attempts only within its own system. During Phase 1, 3DS 1.0 and 2.0 run in parallel. PNB's block rule says 3 wrong OTPs = soft block. But the two systems had separate counters.
THE FRAUD EXPLOIT
A fraudster enters 2 wrong OTPs on a 3DS 1.0 merchant. Switches to a 3DS 2.0 merchant. Gets 3 fresh attempts. PNB's primary fraud control — the block rule — never triggers. The bank is fully exposed.
WHAT I DID
Documented the exact fraud path, got engineering to build a unified counter spanning both systems. Counter increments regardless of which system the wrong OTP occurs on. Validated this as the highest-priority test case in UAT.
How I Convinced Engineering to Build It
"Engineering's initial view was: low probability. My response was: low-probability fraud exploits become high-probability the moment fraudsters discover them — and they always do. I walked through the exact exploit step by step, then made it structural: 'PNB's hard block threshold requires 3 soft blocks in 30 days. Without a shared counter, a fraudster resets after every 2 attempts by switching systems — the hard block never triggers. Your most important fraud control is effectively disabled during Phase 1.' That framing got immediate sign-off. The BA's job is to make risk concrete enough to drive decisions."

📄 FSD — The Blueprint Everyone Built From

~3 weeks. Output: FSD v1.0 — single source of truth for Dev, QA, UAT, and PNB.

The FSD was not a summary of what the platform does. It was a specification of exactly how PNB's implementation should behave — every state, every transition, every error code, every API field. Dev built to it. QA tested against it. Any deviation was a defect.

WHAT MADE THIS FSD UNUSUAL — THE OTP BLOCK STATE MACHINE
The block logic had to be specified as a formal state machine — not a bullet list — because the transitions depended on conditions across two systems and time windows. I documented five states explicitly: ACTIVE → SOFT_BLOCKED (counter=3, resets at midnight) → HARD_BLOCKED (3 soft blocks in 30 rolling days, manual CSR reset only) → UNBLOCKED BY CSR (counter resets to 0). Every transition trigger, reset condition, and error message was specified. QA wrote test cases directly from this. There was no room for interpretation.
  • Bank switch API spec: Endpoint structure, HTTPS/JSON, timeout = 2,000ms, mandatory fields, error codes (E-BS-001 card not found, E-BS-002 timeout, E-BS-003 mobile missing, E-BS-004 malformed JSON), fallback behaviour per error code
  • Cross-system counter spec (the custom logic): Counter stored in shared database layer, not in ACCOSA® or 3DS 1.0 separately; increments on any wrong OTP regardless of source system; soft block triggers when combined counter = 3
  • Version control: Every post-sign-off change required a CR. FSD maintained a change log — date, change, reason, approved by. Dev built to the version number on every JIRA story.
04
⚙️ Dev and QA Coordination — What BA Does During the Build Phase

~6 weeks. Output: Dev completes build, QA completes SIT — all against the FSD.

During dev I wasn't idle — I was preventing the most expensive kind of rework. Three things I focused on: clarity, deviation detection, and dependency tracking.

  • Acceptance criteria quality: Every JIRA story had a happy path, at least 2 failure paths, and specific values — not "the OTP should expire" but "OTP expires at exactly 300 seconds from generation; any input after 300 seconds returns error E-OTP-003"
  • Deviation caught in design review: Dev proposed resetting the block counter on a successful authentication, not just on CSR unblock. I flagged this immediately — a cardholder could complete one transaction to reset their wrong attempt count, then resume fraudulent attempts. The FSD said CSR-only reset. This was corrected before a single line of code was written.
  • Key loading tracker: MC Transport Keys, IAV, SPA-2, VISA ZCMK, CAK — each tracked with request date, expected date, actual receipt date, loaded date, verified date. Not my job to load the keys — my job to know the status of every key at every moment because any one missing key blocks go-live.

I ran UAT differently from SIT. SIT is script-based — QA follows steps, checks outputs. UAT is scenario-based — PNB's operations team were given real business situations, not a test script. "A cardholder calls saying they can't complete an online purchase — find them and unblock them." This tests the system the way it will actually be used.

THE CRITICAL UAT SCENARIO — CROSS-SYSTEM COUNTER
PNB's security team personally tested this. They entered wrong OTPs on a 3DS 1.0 test merchant. Then initiated a transaction on a 3DS 2.0 merchant. The combined counter correctly carried over — block triggered on the 3rd wrong OTP regardless of which system it happened on. This was the highest-stakes test in the project. When it passed, the security lead's comment was "we didn't think to check this ourselves." That is the value of a BA who does proper GAP analysis.

Q: "The cross-system counter sounds like an edge case. Was it really worth the engineering effort?"
  • It was not an edge case — it was a broken primary fraud control. The difference matters.
  • PNB's hard block is their last line of fraud defence on OTP-based authentication. If a fraudster can reset their attempt count by switching systems, the hard block never triggers — not occasionally, but systematically. Every fraudster who knows the exploit gets unlimited OTP attempts.
  • I made this argument to engineering using their own language: "The effort to fix this is a one-time cost. The alternative is a permanent fraud bypass that works for every fraudster, every day, for as long as Phase 1 runs." Engineering approved it the same day.

Q: "What would you do differently on PHNB if you started again today?"
  • I would establish a measurable frictionless baseline during SIT — not just verify that RBA routing worked correctly, but record the actual frictionless vs challenge split in the test environment.
  • Post go-live, PNB asked whether their 65% frictionless rate was healthy or below expectations. I had to answer from platform benchmarks, not PNB-specific SIT data. A proper SIT baseline would have made that question answerable with confidence.

Scope Creep Management
How you prevent, identify and manage scope creep — with a real example
🔒 Scope
📖
Definition & Framework
"Scope creep is when new requirements are added to a project without corresponding changes to timeline, resources, or budget — often informally, without formal approval. As a BA, I'm the first line of defence against it."
Prevention — Before Scope Creep Happens
  • Define scope clearly in the BRD with explicit In-Scope / Out-of-Scope sections — and get formal sign-off
  • Use the Out-of-Scope section as a scope boundary reference throughout delivery
  • Conduct a scope walkthrough with all stakeholders at project kickoff — no ambiguity at the start
  • Document every assumption — unclarified assumptions become scope disputes later
Response — When Scope Creep Appears
  • Acknowledge the request positively — never dismiss it outright
  • Log it as a Change Request — formalise immediately
  • Assess impact: effort, timeline, dependencies, and budget implications
  • Present options: include now (with timeline/cost impact), defer to Phase 2, or decline
  • Escalate to project manager and stakeholders for joint decision
  • Document the decision — never verbal-only agreements on scope
⚡
Real Example — PHNB Scope Creep (STAR)
🅢 Situation
Mid-implementation, PNB's fraud team requested adding an IVR (Interactive Voice Response) authentication option — where cardholders could receive and confirm OTP via an automated phone call instead of SMS only. This was not in the original BRD scope (explicitly marked "Not Applicable" in the spec).
🅣 Task
Respond to the request professionally without disrupting the agreed go-live timeline or creating resentment with the bank's fraud team.
🅐 Action
I thanked the fraud team for raising it, confirmed its value, but immediately referenced the signed BRD scope showing IVR was marked Out-of-Scope. I logged it as a formal Change Request, worked with the engineering lead to estimate the effort (an additional 3-week development cycle), and presented two options to the project steering committee: implement in Phase 2 post go-live, or delay go-live by 3 weeks. I also validated that IVR was not required for Phase 1 regulatory compliance.
🅡 Result
Steering committee agreed to defer IVR to Phase 2. Phase 1 go-live proceeded on time. IVR was logged as a confirmed Phase 2 deliverable, and PNB's fraud team was satisfied that it hadn't been dismissed — just sequenced correctly.

PRIORITY QUESTION — 5 Clients, 5 Different Requirements, Same Product: How Do You Handle It as a BA?
⚠️ Why Interviewers Ask This
This is a Senior BA / Implementation Consultant trap question. They want to see: Can you manage conflicting stakeholder demands without losing product integrity? Do you know the difference between a configuration and a customisation? Can you prioritise and say no with a reason? Do you have a governance model — or do you just say yes to everyone? Wrong answer: "I would gather all requirements and prioritise them." Right answer: A structured framework you've actually used.
🎯 Your Opening — Speak This Exactly
"This is a situation I dealt with directly — at PayU-Wibmo we had 90+ banks on the same 3DS authentication platform, each with different OTP policies, block rules, and SLA requirements. My approach to 5 clients with 5 different requirements for the same product has four layers: understand the requirement deeply before accepting it, classify it as Core, Configurable, or Custom, find the common denominator through structured workshops, and govern changes so no client can silently fragment the product. Let me walk you through how I'd apply this."
Step 1 — Understand the Requirement Before You Accept It

"The first mistake a BA makes is writing down what the client says they want instead of what they actually need. When 5 clients give you 5 different requirements, my first question to each of them is always: 'What problem are you trying to solve?' — not 'what feature do you want?'"

✅ Real Example from PHNB
PNB said: "We want the OTP block counter to reset every time the customer tries again." What they actually needed: prevent fraud without permanently blocking legitimate customers. The solution was a soft-block (day-level reset) + hard-block (30-day rolling) — not a simple reset. If I had just written their stated requirement, the fraud protection would have been broken.
5 Questions to Ask Each Client First
  • What problem does this solve? (business outcome, not feature description)
  • What happens today without this? (pain point validation)
  • Who is affected if this isn't delivered? (real priority signal)
  • Is there a regulatory or compliance driver? (non-negotiable vs preference)
  • What would 'good enough' look like? (often reveals flexibility they didn't mention)
Step 2 — Classify Every Requirement: Core / Configurable / Custom

After the workshop, if you still have genuinely conflicting requirements — requirements that cannot both be satisfied — here is the prioritisation framework:

"[Client name], your requirement is valid and it's on our roadmap. ⚠️ Never Do This
Never tell one client "another client asked for the opposite." Never make a client feel ranked against other clients. Frame all prioritisation decisions as product roadmap decisions — not client vs client competition.
Step 5 — Govern Changes So the Product Doesn't Fragment

The most dangerous moment in a multi-client implementation isn't during discovery — it's 6 weeks into delivery when Client 3 calls your engineering lead directly and asks for "a small change." Without governance, that call creates an undocumented custom build that breaks 2 other clients at the next release.

Change Control Board (CCB) — Structure
  • Members: BA (chair) + Product Owner + Engineering Lead + 1 representative from key clients
  • Triggers: Any new requirement OR change to existing requirement OR scope addition from any client
  • Process: Change Request (CR) raised → impact analysis by BA → CCB review → approve/defer/reject → communicate to all clients
  • Rule: No engineering work starts on any change without a logged CCB decision. Full stop.

Impact on timeline: +2 weeks if delivered as config feature

CCB Decision: Approved — Sprint 6. Owner: BA. Inform all clients.

Follow-Up Questions — Be Ready For These
Q: "What if 2 clients have directly opposing requirements?"
"If two clients need mutually exclusive behaviour on the same feature, I present the product owner with a clear options analysis: Option A satisfies Client 1 — here's what Client 2 loses. "That's a commercial escalation, not a BA decision. My job is to give leadership the complete picture: the cost of building it (dev effort, maintenance debt, regression risk to other clients), the commercial risk of not building it (losing that client), and whether there's a configurable middle ground. "Mid-sprint requirement changes are a governance failure, not a client problem. If a client is changing requirements mid-sprint it means either: the requirements weren't fully understood before sprint start, or there's no change control process in place. My response: freeze the current sprint scope, log the change as a CR, assess impact, and schedule it for the next sprint. I explain to the client: 'Your change is valid and we'll deliver it — but changing it now would break the current sprint's delivery for you and potentially for other clients too. Here's when we can do it properly.' Most clients accept this when they understand the trade-off."

Step 2 — Estimation Techniques
Technique 1 — Analogous Estimation (Most Used in Implementation Consulting)
  • What it is: Using historical data from similar past projects as the baseline estimate
  • How I use it: "We onboarded 90+ banks to the 3DS platform at PayU-Wibmo. A standard bank onboarding with no custom development took 8–10 weeks. A bank with custom OTP logic took 14–16 weeks. PHNB had one critical custom requirement — I estimated 18–20 weeks."
  • Best for: SaaS platform implementations where you have repeat delivery patterns
  • Risk: Only valid when the new project is genuinely similar — must account for differences explicitly
Technique 2 — Three-Point Estimation (T-shirt or PERT)
  • What it is: Estimate three scenarios — Optimistic (O), Most Likely (M), Pessimistic (P) — then calculate weighted average: E = (O + 4M + P) / 6
  • How I use it: For components with high uncertainty — e.g. API integration with a third-party payout partner whose documentation is incomplete
  • Example (Ria Money payout partner API): O = 2 weeks (clean docs, responsive partner), M = 3.5 weeks (some back-and-forth), P = 6 weeks (poor docs, multiple revision cycles). E = (2 + 14 + 6) / 6 = 3.7 weeks. I'd plan for 4 weeks and flag the risk.
  • Best for: Integration work, third-party dependencies, regulatory compliance items
Technique 3 — Story Point / Relative Estimation (Agile context)
  • What it is: Estimating effort relative to a known baseline story, using Fibonacci scale (1, 2, 3, 5, 8, 13)
  • How I use it: In Agile sprints — e.g. "OTP generation flow = 3 points (simple API call + DB write). Cross-system counter sync = 13 points (complex, spans two systems, new logic)."
  • BA's role in story point estimation: Write the acceptance criteria precisely enough that Dev can estimate without asking questions. Vague acceptance criteria = inflated estimates.
  • Velocity-based planning: Once team velocity is known (e.g. 30 points/sprint), total story points ÷ velocity = number of sprints needed
  • Best for: Sprint planning within an Agile delivery track
Technique 4 — Function Point / Feature-Based Estimation
  • What it is: Counting measurable functional units — screens, APIs, reports, workflows — and multiplying by average effort per unit type
  • Example: PHNB FSD had: 4 main screens (OTP page, block message, timeout, CSR dashboard) × 3 days/screen = 12 dev days. 2 API integrations (bank switch, card network DS) × 5 days/API = 10 dev days. 3 background processes (RBA scoring, OTP generation, block counter) × 4 days = 12 dev days. Total dev = ~34 days.
  • Best for: Early-stage commercial scoping where you need a defendable number for the SoW
  • BA's job: Count the features accurately — if you miss a feature at scoping, the estimate is wrong and the SoW is under-priced

Interview Q&A — Effort Estimation
Q: "How do you estimate effort when the client's requirements are not fully clear yet?"
  • Provide a range estimate with named assumptions — "Based on what we know today, this is 10–14 weeks. The range will narrow once we complete discovery and confirm the integration points."
  • Never provide a single-point estimate on unclear scope — it creates a false commitment that will be held against you later
  • Document assumptions explicitly: "This estimate assumes standard platform configuration. If custom OTP logic is required, add 3–4 weeks."
  • Schedule a scope confirmation checkpoint — agree that the estimate will be revisited and confirmed after the discovery workshop
Q: "Your estimate was wrong on a project. How did you handle it?"
  • Acknowledged it early — the moment I saw the estimate was going to be missed, I raised it with the PM and documented the reason: the bank switch API spec was incomplete at scoping time and required 3 additional revision cycles with PNB's IT team
  • Presented a revised estimate with a root cause and corrective action: updated the integration estimation template to require API documentation as a mandatory input before scoping is finalised
  • Did not absorb the overrun silently — hidden overruns damage trust far more than transparent ones with a clear explanation
  • The lesson: estimation is a living document, not a one-time activity. Re-estimate at every phase gate when new information is available.
Q: "How did you manage effort estimation when working on multiple client implementations simultaneously?"
  • At PayU-Wibmo, managing 90+ bank onboardings simultaneously required a tiered effort model: Standard onboarding (no custom dev) = template-based estimate (~8 weeks). Standard + config changes = base + delta (~10–12 weeks). Custom development = full bottom-up estimate (14–20 weeks).
  • Maintained a capacity tracker — total BA hours available per week vs committed hours across all active projects — escalated to management when capacity was at risk
  • Used a phased prioritisation approach: clients in go-live week get highest BA attention; clients in discovery get scheduled slots; clients in UAT get async defect management where possible
  • The key insight: effort estimation and capacity planning are the same problem viewed from different angles — estimate effort accurately, and capacity conflicts become visible early enough to resolve
Senior BA — Hard Questions
Difficult & tricky questions specifically asked at Senior BA level interviews
🎯 Senior BA
✅ Why These Questions Are Hard
Senior BA interviews test judgment, not just knowledge. They want to see you navigate ambiguity, push back on stakeholders, manage conflict, and lead without authority. Don't just define terms — show you've lived them.
⚡
Requirements & Analysis
A stakeholder says "I want everything done in Sprint 1." How do you respond? +
"I'd acknowledge that they want to move fast — that's a good sign of urgency. But I'd walk them through a prioritisation exercise immediately. I use MoSCoW: which of these are Must-Haves for the business outcome you're trying to achieve? Often what appears as 'everything' is actually 3 Must-Haves and 10 Should-Haves. Once we separate those, the conversation shifts from 'do everything' to 'let's sequence to maximise business value early.' In PHNB, Phase 1 and Phase 2 was exactly this — business-critical BINs in Phase 1, remaining features in Phase 2."
How do you write requirements that can't be misinterpreted by engineering? +
"I use the 'System SHALL' format with measurable, specific criteria — no ambiguous words like 'should', 'may', or 'fast'. A good requirement has a clear trigger, action, and expected outcome. For example — bad: 'OTP should expire soon.' Good: 'The system SHALL invalidate the OTP exactly 300 seconds from generation timestamp, regardless of whether it has been used.' I also run a peer review with the engineering lead and QA before finalising any FSD — if they can't write a test case from my requirement, it's not specific enough."
What do you do when you discover a requirement conflict between two signed-off documents? +
"First I document the conflict precisely — which document, which clause, what the contradiction is. Then I raise it with both stakeholder groups and the project manager transparently — not as a problem but as 'I need a decision to proceed.' I facilitate a session to align, referencing the original business objective in the BRD to adjudicate. The resolved decision gets formally documented as a CR or BRD amendment with both parties signing off. I've found that conflicts between signed documents usually stem from different teams having different assumptions at the time of signing — the fix is alignment, not blame."
How do you handle a requirement that is technically impossible? +
"I first verify it is actually impossible vs. just expensive or complex — those are very different things. I get the engineering team to explain exactly what constraint makes it impossible. Once confirmed, I go back to the business objective behind the requirement — not the requirement itself. Usually there's an alternative way to achieve the same outcome. I present 2–3 alternative approaches with trade-offs and let the stakeholder decide. For example: if real-time fraud scoring is not possible due to latency, near-real-time with a 2-second delay may achieve the same fraud prevention objective."
A business user keeps changing requirements after sign-off. What do you do? +
"I address the root cause first — usually it's either unclear requirements at sign-off, or the business user didn't fully understand what they were signing. First change: I log it as a formal CR, assess impact, and communicate cost/time implications clearly. If it keeps happening, I set up a weekly change review board so changes are batched and evaluated together rather than ad hoc. I also revisit the requirements gathering process — if we're getting frequent changes post sign-off, we may need more walkthroughs and prototypes before sign-off in future cycles. The goal is to make change management feel structured and fair, not punitive."
🔄
Process & Delivery
How do you prioritise when everything is a priority? +
"When everyone says their requirement is Priority 1, I use three lenses: business value (what revenue, cost, risk, or compliance impact does this have?), urgency (is there a deadline or regulatory driver?), and dependency (does anything else block on this?). I then facilitate a prioritisation workshop using MoSCoW and ask stakeholders to make trade-offs explicit — if we do X this sprint, Y moves to next sprint. People are more willing to deprioritise when they see the trade-off clearly and trust that Y will definitely happen later. I always document the agreed prioritisation so there's no revisiting it informally."
Tell me about a time you pushed back on a deadline that was unrealistic. +
"In a project where a bank wanted a full 3DS 2.0 implementation in 4 weeks — a timeline that would typically take 8–10 weeks — I didn't just say no. I broke down the work into a detailed task list with effort estimates and presented it visually with the critical path. I showed specifically which steps would be skipped or rushed and what risks that created — for example, skipping a full UAT cycle meant live bugs reaching production on a fraud-sensitive system. I then presented a risk-based phased approach: core functionality in 6 weeks with a limited go-live, full feature set in Week 10. The client accepted the phased approach because I gave them facts, not just pushback."
What's the difference between a good BA and a great BA? +
"A good BA captures what the stakeholder asks for. A great BA understands what the stakeholder actually needs and proactively identifies gaps, risks, and dependencies they haven't thought of. In PHNB, no one asked me to think about the cross-system OTP counter problem — I identified that gap myself during GAP analysis because I was thinking about how the two systems would interact during parallel running. A great BA is also a connector: between business and engineering, between current reality and future state, and between what's been agreed and what's actually being built. And critically — a great BA knows when to escalate vs. when to solve it at ground level."
How do you manage a project where the client doesn't know what they want? +
"This is very common. I use a structured discovery approach: start with the problem statement, not the solution. I ask 'what's broken today?' and 'what does success look like 6 months from now?' rather than 'what features do you want?' I use techniques like process walkthrough (show me how you do it today), pain point mapping, and in some cases a simple prototype or wireframe to make abstract requirements tangible. Once they can see something, they can react to it — and their reactions give you the real requirements. I also time-box discovery so it doesn't spiral. A clear problem statement and measurable success criteria is all you need to start building."
🧠
Senior-Level Judgment Questions
Have you ever disagreed with a project manager's decision? What did you do? +
"Yes. In one case the PM wanted to skip the full UAT cycle to meet a deadline, moving straight from dev to go-live. As BA I owned the quality of the requirements and I believed this was too risky — particularly for a payments system where a bug could mean failed transactions at scale. I raised my concern privately with the PM first, presenting specific test scenarios that hadn't been validated. When the PM still wanted to proceed, I escalated formally to the project steering committee — not as a conflict, but as 'I need the committee to be aware of this risk and make an informed decision.' The committee agreed to a 3-day accelerated UAT cycle as a compromise. It's important to escalate through the right channel, with data, not emotion."
How do you ensure business requirements don't get lost in technical translation? +
"I use a Requirement Traceability Matrix throughout the project — every BRD requirement maps to at least one FSD specification, which maps to at least one user story, which maps to at least one UAT test case. If any requirement has no test case, it hasn't been built. I also attend sprint reviews and UAT from the business lens — not just checking if it works technically, but checking if it meets the original business intent. In PHNB, I ran each UAT scenario against the original BRD requirement to confirm the technical implementation matched what PNB's fraud team had asked for — not just what engineering had built."
What's your process when you join a project that is already in trouble? +
"First I do a rapid assessment: I review all existing documents — requirements, FSD, UAT results, open defects, and stakeholder communication logs. I identify the root cause of the trouble — is it unclear requirements, scope creep, technical blockers, or stakeholder misalignment? I don't start suggesting solutions until I understand the actual problem. Then I facilitate a reset workshop with key stakeholders — present what I've found, agree on the current state, and define a clear forward plan with owners and dates for every action. The most important thing is to restore confidence through structure and transparency — not promises."
How do you measure the success of a BA's work? +
"Three ways: first, defect density in UAT — if requirements are clear and complete, engineering builds the right thing and UAT defects are low. A spike in UAT defects often traces back to ambiguous requirements. Second, stakeholder satisfaction — did the delivered solution match what business stakeholders expected? Surprises at UAT or go-live mean requirements weren't communicated clearly. Third, post-go-live stability — how many production incidents in the first 30 days were due to missed requirements vs genuine edge cases? I track this personally because it's the most honest feedback loop on the quality of my BA work."

✏️ Phase 3 — Test Case Design
  • Test design techniques applied:
  • Equivalence Partitioning: Ria PIN field — Valid (11-digit numeric), Invalid-short (<11 digits), Invalid-alpha (contains letters), Invalid-claimed (already used), Invalid-nonexistent. Each partition = one test case.
  • Boundary Value Analysis: Amount field — test at ₹1 (min), ₹2,500 USD equivalent (RBI limit), ₹2,501 (above limit), ₹0 (zero), negative values. OTP timer — test at 0:01 remaining (boundary), 0:00 (expired). Account balance — test at exactly permissible limit, 1 rupee above limit.
  • Decision Table Testing: KYC routing has 7 customer types × 2 account states (with/without savings) × 3 payout modes = complex branching logic. Built a decision table mapping all combinations to expected routing outcomes. This ensured no branch was missed.
  • State Transition Testing: Transaction status lifecycle — INITIATED → PENDING → SUCCESS / FAILURE. OTP states: GENERATED → VALID → EXPIRED / USED / REGENERATED. Card/CIF states: ACTIVE → BLOCKED (AML). Each transition is a test case.
  • Error Guessing: Based on domain knowledge — tested: same Ria PIN used twice in same session, biometric attempted before sender name confirmation, Form 60 submitted with blank income fields, IFSC code with correct format but non-existent bank, beneficiary account number with correct check digit but wrong bank.
  • Output: 47 test cases documented in test management format: Test Case ID | Module | Scenario | Preconditions | Steps | Test Data | Expected Result | Actual Result | Status | Defect ID (if any).

What BA Does in Functional Testing

  • Execute in UAT — facilitate business users through scenarios

SIT — System Integration Testing
"SIT tests the complete integrated system — all modules together — before business users see it. Executed by QA in a controlled test environment. BA provides test cases, clarifies requirements, reviews results."
SIT Characteristics
  • Environment: Test/Staging — never production
  • Executor: QA team (BA supports)
  • Data: Synthetic test data
  • Entry criteria: Build deployed, smoke test passed, test data ready
  • Exit criteria: All P1/P2 closed, 100% coverage, QA sign-off
  • Cycles: Initial → defect retest → regression
BA's Role in SIT
  • Provide test cases mapped to requirements (RTM)
  • Clarify requirement ambiguities QA raises
  • Triage defects — genuine defect vs misunderstood requirement?
  • Prioritise — P1 (go-live blocker) vs P3 (cosmetic)
  • Confirm fix matches FSD before closing defect

Regression Testing
"Regression testing ensures new code changes haven't broken previously working functionality. Run after every defect fix cycle and before every release."
When to Run Regression
  • After every defect fix batch in SIT
  • After every sprint release (Agile)
  • Post-SIT before UAT escalation
  • After any production hotfix
  • Before major go-live
Types
  • Full regression — all test cases, major releases
  • Selective/Partial — impacted modules only
  • Sanity — just the fixed area, quickest
BA's Role
  • Impact analysis: Which other requirements could the fix affect?
  • Define scope: Which test cases to re-run — prevents over and under testing
  • RTM update: Updated after every regression cycle

📄 BRD / FSD
🎯 The One-Line Difference — Say This First
"A BRD answers what the business needs and why. An FSD answers how the system will behave to meet those needs. BRD is business language for business sign-off. FSD is functional specification for engineering, QA, and delivery teams. As a BA, I own both — and I ensure full traceability between them so nothing gets lost in translation."
📋
PART 1 — BRD: Business Requirements Document
"The BRD is the contract between the business and the delivery team. It captures what the business wants to achieve — not how the system will do it. I write the BRD after stakeholder workshops and discovery, and it becomes the foundation document that everything else is built on. BRD sign-off is mandatory before any FSD or development work begins."
🔷 What the BA Does to Create a BRD
1
Stakeholder Identification & RACI
Identify all stakeholders — business owners, IT leads, compliance, operations, end-users. Define a RACI (Responsible, Accountable, Consulted, Informed) for the project. For PHNB: PNB Business Team (Accountable), Wibmo BA & Engineering (Responsible), EMVCo/Network Compliance (Consulted), PNB IT (Informed).
Output: Stakeholder Register + RACI Matrix
2
Discovery & Requirement Elicitation
Conduct structured workshops — interviews, JAD sessions, observation, document analysis. Techniques: 5 Whys (understand root cause behind requirements), brainstorming, prototyping, use case walkthroughs. For PHNB: workshops with PNB cards team to understand their OTP policy, fraud thresholds, card types, BIN structure, SMS vendor integration, and compliance requirements.
Output: Workshop notes, elicitation artifacts, draft requirement list
3
As-Is Process Analysis
Document the current state — what exists today, how it works, what pain points it has. For PHNB: current 3DS 1.0 ACS setup, current OTP flow, card base structure, existing BIN routing, current fraud rates, and why 3DS 1.0 needs replacement (mobile failures, EMVCo mandate, liability exposure).
Output: As-Is Process Map, Current State Assessment
4
GAP Analysis (As-Is → To-Be)
Compare current state against target state. Identify what the platform already supports (configurable), what needs to be built (custom), and what is outside scope. For PHNB: ACCOSA® IVS supports 3DS 2.0 natively — GAP was the cross-system OTP block counter spanning 3DS 1.0 and 2.0, which needed a custom engineering solution. Documented as a named GAP with business justification.
Output: GAP Analysis Table — As-Is | To-Be | GAP | Solution | Owner
5
Requirement Structuring & Prioritisation
Organise elicited requirements into structured categories — Business Requirements, Functional Requirements (high-level), Non-Functional Requirements, Constraints, Assumptions. Apply MoSCoW prioritisation with business stakeholders. Every requirement gets an ID for traceability (BR-001, BR-002…).
Output: Structured Requirements List with MoSCoW priority and IDs
6
Validation & Business Sign-Off
Review BRD with all stakeholders in a formal walkthrough. Resolve conflicts. Obtain written sign-off from business owner. No FSD or development work starts until BRD is signed. After sign-off, changes go through Change Request process — no informal updates.
Output: Signed BRD + Change Request process activated

✅ BA Golden Rule for BRD
"Every requirement in the BRD must be: Testable (can I write a test case for it?), Traceable (does it link to a business objective?), Unambiguous (does every stakeholder read it the same way?), and Feasible (can engineering actually build it within constraints?). If a requirement fails any of these, it goes back to the stakeholder for clarification before BRD sign-off."
📐
PART 2 — FSD: Functional Specification Document
"The FSD translates BRD business requirements into precise functional specifications that engineering, QA, and UX can build against. Every screen, every business rule, every integration point, every error condition — fully defined. If it isn't in the FSD, it will not be built correctly. The FSD is the BA's most critical deliverable."
🔷 How BA Writes an FSD — The Process
  • Start from BRD: Every FSD section traces back to a BRD requirement ID. No orphan specifications.
  • Decompose business requirements: BR-002 "OTP block counter spans both systems" → FSD section 3.4 with exact counter logic, reset rules, error codes, and API behaviour.
  • Use wireframes as reference: FSD describes the behaviour behind every screen — field validations, navigation rules, error messages, conditional logic.
  • Write in active voice: "System SHALL" (mandatory), "System SHOULD" (recommended), "System MAY" (optional). No passive or ambiguous language.
  • Define every error state: For each business rule — what happens when it's violated? Exact error message text, error code, and recovery path.
  • Get engineering review: FSD walkthrough with dev team — confirm feasibility, identify technical constraints, capture agreed changes.
  • Get QA review: QA reads FSD to confirm every requirement is testable. QA starts writing test cases from the FSD during this phase.
  • Version control: Every FSD change logged with reason, impact, approver. Any change after engineering starts = formal Change Request.

PHNB FSD — OTP Block Rule (FR-3.5.3)
FR-3.5.3 — OTP Block Counter Rules

Business Rule: A cardholder who enters incorrect OTP 3 consecutive times shall be soft-blocked for the remainder of that calendar day.

Pre-condition: Cardholder has received valid OTP and is on the challenge screen.

Trigger: Cardholder submits incorrect OTP.

System SHALL:
a) Increment incorrect OTP attempt counter by 1.
b) If counter = 1 or 2: display "Incorrect OTP. X attempt(s) remaining."
c) If counter = 3: soft-block card for calendar day. Display "Card temporarily blocked. Contact bank."
d) Counter SHALL span both 3DS 1.0 and 3DS 2.0 systems during Phase 1 parallel running.
e) If cardholder accumulates 3 soft-blocks within 30 calendar days: hard-block card. CSR manual unblock required. Counter resets to 0 on hard-block.

Error Code: E-OTP-003 (Soft Block), E-OTP-004 (Hard Block)

04Business Rules & Validation LogicCritical▼
What It Contains
  • Field-level validation rules: Every input field — mandatory/optional, data type, format, min/max length, allowed characters
  • Business logic rules: Calculation rules, decision rules, constraint rules
  • Conditional rules: IF [condition] THEN [action] ELSE [fallback] — written explicitly for every branch
  • Time-based rules: Timeouts, validity windows, scheduled jobs
  • Compliance rules: Regulatory or network-mandated rules that cannot be deviated from
How to Write Business Rules

Use IF-THEN-ELSE structure. Be precise with numbers — "approximately 5 minutes" is wrong; "300 seconds from OTP generation timestamp" is correct. Never use "normally", "usually", "should generally" — these are untestable.

PHNB Business Rules Examples
BR-RULE-01: OTP Validity
OTP SHALL be valid for exactly 300 seconds (5 minutes) from generation timestamp. After 300 seconds, OTP status = EXPIRED. System SHALL NOT accept expired OTPs regardless of correctness.

BR-RULE-02: OTP Resend
On Resend: System SHALL generate a NEW 6-digit OTP. Previous OTP SHALL be invalidated immediately. Timer SHALL reset to 300 seconds. Maximum resend attempts: [as per bank config].

BR-RULE-03: Page Timeout
If cardholder takes no action on challenge screen for 420 seconds (7 minutes), system SHALL return ARes with status = U (Unable to Authenticate). No OTP consumed.

BR-RULE-04: Frictionless Threshold
IF TRIDENT risk score = LOW THEN authenticate frictionless (ARes Y, no challenge). IF TRIDENT risk score = MEDIUM OR HIGH THEN initiate OTP challenge flow.

STAR Answer — PHNB Testing (For Interview)
🅢 Situation
Philippines National Bank was implementing EMV 3DS 2.0 for 1.35 million cards — Debit, Credit, and Prepaid on both Mastercard and VISA. The most critical requirement was a cross-system OTP block counter spanning both 3DS 1.0 and 3DS 2.0 during Phase 1 parallel running — a non-standard engineering solution with no existing test precedent.
🅣 Task
Design and execute the full SIT test suite — 47 test cases across 10 modules — coordinate defect resolution, lead UAT with PNB's operations and cards team, and deliver a go-live recommendation with zero open P1/P2 defects.
🅐 Action
Built the test suite using Equivalence Partitioning, Boundary Value Analysis (OTP timer boundaries at 299s/300s, page timeout at 420s), Decision Tables for RBA routing and card type combinations, State Transition testing for block lifecycle (ACTIVE → SOFT_BLOCKED → HARD_BLOCKED → UNBLOCKED), and targeted Error Guessing for the cross-system counter and API failure paths. Found 6 defects in SIT — including 2 P1s: a null pointer crash on missing mobile number and a counter reset logic error that would have allowed fraud bypass. Both fixed and verified in Cycle 2. UAT used scenario-based testing with real PNB card data, uncovering 3 additional real-world issues SIT's synthetic data hadn't caught — SMS delay UX, international roaming number format mismatch, and inactive card search filter.
🅡 Result
47 SIT test cases, 3 cycles, 6 defects found and all closed. 10 UAT scenarios, 3 UAT defects resolved. Formal PNB Operations Head sign-off. Phase 1 go-live on schedule. Zero P1/P2 production incidents in the first 30 days. The cross-system counter — the highest-risk requirement — worked exactly as specified from day one of production.
